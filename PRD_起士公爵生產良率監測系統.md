---

## 1. 產品願景

建立一套**手機優先、現場即時填寫**的生產品質監測系統，讓起士公爵烘焙團隊能在每個關鍵工站即時記錄品質數據，藉此實現：

- 🎯 **即時監控** — 異常發生時立即通知班長/主管
- 📊 **數據驅動** — 以客觀數據取代主觀經驗判斷
- 👥 **人員培訓** — 追蹤個人與團隊績效，針對性訓練
- 🔄 **持續改善** — 長期趨勢分析，找出系統性問題

## 4. 功能需求 — 11 個關鍵監控項目

### 4.1 均質站（2 項）

| 項目 | 標準值 | 類型 | 異常處理 |
|------|--------|------|---------|
| 乳酪溫度 | 13℃ ~ 15℃ | 數值輸入 | 偏離 → 通知班長 |
| 均質時間 | 8'30" ~ 9'00" | 分秒輸入 | 偏離 → 通知班長 |

### 4.2 蛋白霜站（3 項）

| 項目 | 標準值 | 類型 | 異常處理 |
|------|--------|------|---------|
| 快速打發時間 | 5'20"（固定） | 分秒輸入 | 偏離 → 通知班長 |
| 蛋白霜比重 | 68g ~ 70g | 數值輸入 | 偏離 → 通知班長 |
| 拌合次數 | 55 ~ 60 下 | 數值輸入 | 偏離 → 通知班長 |

### 4.3 整面站（1 項）

| 項目 | 標準值 | 類型 | 異常處理 |
|------|--------|------|---------|
| 筷子切泡 | 約 3 圈 | 下拉選擇 | 非標準 → 通知班長 |

### 4.4 烤箱站（2 項，4 個欄位）

| 項目 | 標準值 | 類型 | 異常處理 |
|------|--------|------|---------|
| 烘烤溫度 | 上火 230℃ / 下火 120℃ | 雙欄位 | 偏離±5℃ → 通知班長 |
| 烘烤時間 | 進爐 25分 + 轉盤後 35~40分 | 雙欄位 | 偏離 → 通知班長 |

### 4.5 出爐品判站（3 項）

| 項目 | 標準值 | 類型 | 異常處理 |
|------|--------|------|---------|
| 出爐熟度 | 觸摸後中心回彈 | 下拉選擇 | 完全不回彈 → 🔴 嚴重，通知主管 |
| 出爐顏色 | 色卡 4 ~ 6 | 數值輸入 | 偏離 → 通知班長 |
| 外觀不良 | 無 | 多選按鈕 | 有勾選 → 🔴 嚴重，通知主管 |

**外觀不良選項：** 表面有泡泡、裂開、過黑（上色過深）、烤不熟

## 5. 批次結構

### 5.1 工單格式

每日由 Google Sheets 工單產生，結構為：

```
日期 → 輪次 (1~N) → 烤箱 (A/B/C) → 口味
```

**批次編號規則：** `{輪次}-{烤箱}` 例：`1-A`, `2-B`, `3-C`

### 5.2 範例（2026/4/14 工單）

| 輪次 | 烤箱 A | 烤箱 B | 烤箱 C |
|------|--------|--------|--------|
| 第 1 輪 | 天使 | 原味 | 原味 |
| 第 2 輪 | 楓糖 | 玫荔 | 玫荔 |
| 第 3 輪 | 原巴 | — | — |

**口味列表：** 天使、原味、楓糖、玫荔、原巴、生巧

### 5.3 記錄頻率

- 每個批次（每爐）在每個工站各記錄一次
- 出爐時立即記錄，尤其發現不良時要盡快記錄
- 一天一個工單，包含所有輪次

### 6.2 技術選型

| 層 | 技術 | 理由 |
|---|------|------|
| 前端 | 單一 HTML + 內嵌 CSS/JS | 零依賴、任何裝置皆可開啟 |
| 路由 | Hash-based (#均質, #看板) | 不需後端路由支援 |
| 離線快取 | localStorage | 斷網時仍可填寫，上線後同步 |
| 後端 API | Google Apps Script | 免費、無需管理伺服器 |
| 資料庫 | Google Sheets | 免費、主管可直接開 Sheets 查看 |
| 部署 | GitHub Pages | 免費、HTTPS、自動部署 |
| 品牌色 | 起士公爵綠 #3A7D5C | 取自官網 cheeseduke.com.tw |

### 6.3 資料流

```
操作員填寫 → 即時驗證(合格/異常/嚴重)
           → 儲存到 localStorage
           → 自動 POST 到 Apps Script
           → 寫入 Google Sheets
           → 看板/分析頁即時呈現
```

## 8. 數據分析規格

### 8.1 分析頁面提供的報表

| 報表 | 維度 | 說明 |
|------|------|------|
| 概覽 | 日期區間 | 記錄天數、筆數、整體良率、嚴重異常數 |
| 各站良率 | 站別 | 5 站各自的合格率（橫條圖） |
| 人員良率 | 操作人員 | 每位人員的合格率排名 |
| 口味別良率 | 口味 | 各口味的合格率比較 |
| 人員×站交叉 | 人員 × 站 | 矩陣表：誰在哪一站弱 |
| 口味×站交叉 | 口味 × 站 | 矩陣表：哪口味在哪站出問題 |
| 每日趨勢 | 日期 | 每天的良率變化 |
| 異常明細 | 全部 | 所有異常記錄的清單 |

### 8.2 良率計算公式

```
良率 = 合格檢查項目數 / 總檢查項目數 × 100%
```

- 🟢 ≥ 90% 合格（綠色）
- 🟡 70% ~ 89% 需關注（黃色）
- 🔴 < 70% 嚴重（紅色）

## 9. 異常分級與處理

| 等級 | 觸發條件 | 顏色 | 處理方式 |
|------|---------|------|---------|
| ✅ 合格 | 所有項目在標準範圍內 | 🟢 綠色 | 無需處理 |
| ⚠️ 異常 | 有項目超出標準範圍 | 🟡 黃色 | 通知班長/主管確認 |
| 🔴 嚴重 | 出爐熟度「完全不回彈」或有外觀不良 | 🔴 紅色 | 立即通知主管，可能整批報廢或返工 |

## 10. Google Sheets 雲端同步規格

### 10.1 Sheets 結構

**記錄表（Sheet: 記錄）**

| 欄位 | 說明 | 範例 |
|------|------|------|
| 日期 | YYYY/MM/DD | 2026/04/15 |
| 時間戳記 | ISO 8601 | 2026-04-15T08:30:00.000Z |
| 操作人員 | 姓名 | 張宏霖師傅 |
| 工作站 | 站名 | 均質 |
| 批次 | 編號 | 1-A |
| 口味 | 口味名 | 天使 |
| 合格數 | 整數 | 2 |
| 異常數 | 整數 | 0 |
| 嚴重數 | 整數 | 0 |
| 外觀不良 | 逗號分隔 | 泡泡、裂開 |
| 備註 | 文字 | |
| 原始數據 | JSON | {"cheeseTemp":"14",...} |

**工單表（Sheet: 工單）**

| 欄位 | 說明 |
|------|------|
| 日期 | YYYY/MM/DD |
| 批次編號 | 1-A |
| 輪次 | 1 |
| 烤箱 | A |
| 口味 | 天使 |

**人員表（Sheet: 人員）**

| 欄位 | 說明 |
|------|------|
| 姓名 | 張宏霖師傅 |

### 10.2 同步機制

- **寫入時機：** 每次提交記錄後自動同步
- **唯一鍵：** 日期 + 批次 + 工作站（相同組合會更新而非重複新增）
- **離線支援：** 斷網時存在 localStorage，上線後手動同步
- **讀取時機：** 數據分析頁面查詢時從雲端拉取

### 10.3 設定步驟

1. 建立新 Google Sheets（命名為「起士公爵_生產監測資料庫」）
2. 點選 **擴充功能 → Apps Script**
3. 貼上 `google-apps-script.js` 的內容
4. 點選 **部署 → 新增部署作業**
   - 類型：網頁應用程式
   - 執行身分：我
   - 存取權限：所有人
5. 授權並複製 Web App URL
6. 到監測系統 **管理設定 → ☁️ Google Sheets 雲端同步** 貼上 URL
7. 點「儲存並測試連線」確認成功

## 13. 已知限制與風險

| 限制 | 影響 | 緩解方案 |
|------|------|---------|
| localStorage 有容量限制 (5~10MB) | 約可存 10,000+ 筆記錄 | 定期匯出 + 雲端同步 |
| Google Apps Script 有執行時間限制 | 單次請求 6 分鐘 | 分批同步、資料量小可忽略 |
| 無使用者驗證機制 | 任何人知道網址都能填寫 | V3 可加 PIN 碼或 LINE Login |
| 資料依賴瀏覽器 | 清 cache 會遺失本地資料 | 已有雲端同步做備份 |
